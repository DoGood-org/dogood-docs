# DTO і моделі

## Що таке DTO

**DTO** = **Data Transfer Object**.

Це об'єкт, який використовується для **передачі даних між частинами системи**:
- між фронтендом і бекендом
- між контролером і сервісом
- між різними сервісами

DTO не обов'язково дорівнює структурі таблиці в базі даних.

## Навіщо потрібен DTO

DTO потрібен, щоб:
- передавати тільки потрібні поля
- не відкривати внутрішню структуру БД назовні
- уникати зайвих або небезпечних полів у request/response
- явно фіксувати контракт API

## Основна ідея

Одна сутність у системі може мати кілька представлень:

1. **Request DTO**  
   Дані, які фронтенд відправляє на бекенд.

2. **DB Model**  
   Дані, які реально зберігаються в базі даних.

3. **Response DTO**  
   Дані, які бекенд повертає фронтенду.

## Чому не можна використовувати одну й ту ж модель всюди

Тому що:
- не всі поля з БД повинні приходити на фронтенд
- не всі поля, які приходять з фронтенду, зберігаються в БД напряму
- частина полів є службовими або обчислюваними
- частина полів є relations, а не простими колонками таблиці

## Приклад

Для `Task`:

- фронтенд може передати `isOrganization`
- у базі може зберігатися `hostId`
- у відповіді може приходити `host`, `joinedUsers`

Це три різні представлення однієї сутності.

## Правило документації

Для кожної важливої сутності бажано окремо документувати:

- **Request DTO**
- **DB Model**
- **Response DTO**

Це дозволяє уникнути неоднозначності та спрощує роботу і фронтендерів, і бекендерів.
